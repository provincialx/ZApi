# FreeQwenApi Proxy — Agent Brief (Session 2026-06-07)

## Архитектура
FreeQwenApi = браузерный прокси ForgetMeAI. Node.js + Puppeteer.
*   Заменяет веб-аккаунт Qwen Chat на локальный OpenAI-compatible API (`localhost:3264`)
*   Эмулирует вызовы Qwen через JS `fetch` внутри puppeteer-страницы
*   Zed Agent (Browser) → FreeQwenApi → Qwen Chat (через браузер) → ответ обратно
*   Цель: Полноценный агент с tool_calls через интерфейс Zed. Терминальные обёртки исключены.

## Zed.dev + chat.qwen.ai (что узнали)
### Zed
- Rust-based редактор, GPU-accelerated GPUI
- Agentic Editing: встроенные AI-агенты с параллельным выполнением, MCP-серверы
- Edit Prediction через Zeta2 модель. Vim-friendly modal editing
- Any Agent, Any Tool — поддерживает ACP (Agent Communication Protocol), Claude/Codex/OpenCode agents
- Подключается к OpenAI-compatible API: `base_url`, `api_key` в settings

### Qwen Studio (chat.qwen.ai)
- Веб-чат Qwen. Mobile-first, требует скачивания App на десктопе ("Current System does not Support")
- Для прокси не важно — мы стучимся к `/api/v2/chat/completions` через Puppeteer, а не к фронтенду

## Найденные проблемы
### Сессия 1 (решены)
1. **`resolveQwenChatId` без return** → функция возвращала `undefined`, маппинг чатов не сохранялся.
2. **Орфан-импорт посреди файла** (`import { testToken }`) → нарушало структуру JS модуля.
3. **«Потрошное месиво» в логах** (273 сообщения) → бесконечная строка ролей забивала консоль.
4. **Смерть сессий (`chat_not_exist`)** → Qwen удалял старые чаты, прокси использовал кэшированный ID.
5. **Неряшливый лог ошибок** (`undefined`) → `handleApiError` возвращал falsy `{ error: undefined }`.

### Сессия 2 (решены сегодня)
6. **Отсутствие `protocolTimeout` в puppeteer.launch()** → дефолт 180с убивает большие задачи.
7. **Бесконечный hang health-check пула страниц** → `page.evaluate()` без таймаута.
8. **Нет retry при сетевых ошибках goto** → мгновенный фейл на `ERR_CONNECTION_RESET`.
9. **Sliding window рвёт tool_call пары** → `.slice(-40)` мог оставить orphan tool без assistant context.

### Сессия 3 (решены сегодня)

11. **applyToolPrompt не принимал `inAgentLoop`** → agent-loop с tool results получал агрессивный prompt вместо light mode. Фикс: `toolsToLightPrompt()` + 3-й параметр.

### Сессия 4 (решены сегодня)

### Сессия 5 (решено сегодня)

### Сессия 6 (текущая — 2026-06-07 13:xx)
14. **`logWarn is not defined` в routes.js L1498** → ReferenceError crash на streaming пути. Фикс: добавить `logWarn` в import L25.
15. **Qwen возвращает ОБЫЧНЫЙ ТЕКСТ вместо JSON tool_calls** → при `captureToolCalls=true` (stream + tools present), модель пишет "Файл создан..." без `{"tool_calls":[...]}`. parseToolCallJson получает текст, не парсит JSON, returns null.
16. **Double-injection tool prompt в buildPayloadV2** → system_message уже содержал prompt от routes.js (applyToolPrompt вызван ДО sendMessage). buildPayloadV2 вызывал applyToolPrompt ВТОРОЙ раз → дублирование инструкций, возможная путаница модели. Фикс: убран вызов, оставлен debug log.

### Сессия 7 (текущая — 2026-06-07 13:xx)
17. **CAPTURE MODE debug подтвердил: контент = TEXT без JSON** → добавил лог в chat.js L1263 при captureToolCalls. `content type=string, len=275-303, preview="Файл ...создан."`. Никакого `{` в ответе Qwen. Модель полностью игнорирует prompt "respond ONLY with minified JSON".
18. **Tool prompt experiment: reasoning-first FAIL** → попробовал изменить prompt на "напиши рассуждение СНАЧАЛА, затем JSON внизу". Result: модель пишет ТОЛЬКО текст, JSON не генерирует вообще. Вернул агрессивный "respond ONLY with minified JSON" — но даже он НЕ работает при tool history в контексте.
19. **buildStatelessTranscript отправлял сырой JSON-блоб Qwen** → `JSON.stringify(msg.tool_calls)` давал 120+ символов необработанных данных, которые Qwen не парсит и воспринимает как шум. Фикс: заменил на `"Assistant used tools: write_file"` (натуральный язык). Аналогично tool results → `"Tool result (write_file): ..."`.

**ВЫВОД:** Прокси работает корректно (отправляет данные Qwen, получает ответ). Проблема: **Qwen НЕ генерирует JSON tool_calls когда в контексте есть tool history + folded transcript.** Модель видит "Assistant used tools" → "Tool result" и интерпретирует это как завершённый диалог → отвечает текстом вместо вызова инструмента.

Это проблема PROMPT/CONTEXT, не прокси-кода:
1. Qwen не понимает instruction "CALL A TOOL" при наличии tool results в истории
2. Light prompt (inAgentLoop=true) недостаточно агрессивен для повторных вызовов
3. Агрессивный prompt игнорируется — модель предпочитает ответить текстом

**Гипотеза:** проблема на стыке Qwen Chat behavior + agent-loop Zed:
- Прокси отправляет правильный OpenAI-compatible SSE (curl подтверждает)
- Zed Agent получает tool_calls но не исполняет ИЛИ не возвращает result обратно
- Без feedback loop от Zed → Qwen думает что всё сделано → отвечает текстом

## Критический код — как работает сегодня
```
Zed Agent POST /api/chat/completions { messages: [...], tools: [...] }
  → routes.js парсит body, извлекает system message + combinedTools
    → applyToolPrompt(systemMessage, combinedTools, inAgentLoop) добавляет инструмент-схемы к системному промпту (НЕ к Qwen API!)
      → prepareOpenAIMessageInput() → shouldFoldOpenAITranscript() для agent-loop с tool history
        → buildStatelessTranscript(): "User: ...", "Assistant used tools: write_file", "Tool output (write_file): ..."
          → resolveQwenChatId(effectiveChatId, model) → маппинг generated→qwen ID. Если не существует — auto-retry с новым чатом.
            → sendMessage(messageContent, model, qwenChatId, ...) через Puppeteer fetch к /api/v2/chat/completions
              → executeApiRequest() → Qwen отвечает SSE stream (или JSON fallback)
                → если captureToolCalls=true: parseToolCallJson(content) → если JSON найден → OpenAI message.tool_calls format
                  → routes.js возвращает клиенту через buildOpenAIToolResponse() или writeToolCallsSse() (streaming)

ПРОБЛЕМА: при tool history в контексте Qwen иногда отвечает plain text без JSON → parseToolCallJson returns null → 
tool_calls не отправляются, Zed видит пустой response.
```
