# FreeQwenApi Proxy — Agent Brief (Session 2026-06-07)

## Архитектура
FreeQwenApi = браузерный прокси ForgetMeAI. Node.js + Puppeteer.
*   Заменяет веб-аккаунт Qwen Chat на локальный OpenAI-compatible API (`localhost:3264`)
*   Эмулирует вызовы Qwen через JS `fetch` внутри puppeteer-страницы
*   Zed Agent (Browser) → FreeQwenApi → Qwen Chat (через браузер) → ответ обратно
*   Цель: Полноценный агент с tool_calls через интерфейс Zed. Терминальные обёртки исключены.

## Zed.dev + chat.qwen.ai (что узнали)
### Zed
- Rust-based редактор, GPU-accelerated GPUI
- Agentic Editing: встроенные AI-агенты с параллельным выполнением, MCP-серверы
- Edit Prediction через Zeta2 модель. Vim-friendly modal editing
- Any Agent, Any Tool — поддерживает ACP (Agent Communication Protocol), Claude/Codex/OpenCode agents
- Подключается к OpenAI-compatible API: `base_url`, `api_key` в settings

### Qwen Studio (chat.qwen.ai)
- Веб-чат Qwen. Mobile-first, требует скачивания App на десктопе ("Current System does not Support")
- Для прокси не важно — мы стучимся к `/api/v2/chat/completions` через Puppeteer, а не к фронтенду

## Сессия 8 (2026-06-07 — стабилизация tool calling + dedup)

2. Внедрил Idempotency Cache (Dedup) в `routes.js`. Первый запрос кэшируется на 5 секунд. Повторные идентичные запросы от Zed сразу получают SSE-ответ из памяти без дергания Qwen.

## Сессия 10 (текущая — 2026-06-08 — Force Folding)
**О6 решён**: Implemented Force Folding для автоматического сброса контекста Qwen.

**Как работает:**
1. `TOOL_CALL_RESET_THRESHOLD` (по дефолту 8) — после N tool calls, `invalidateModelDefaultChat(model)` инвалидирует чат и добавляет модель в `forceResetModels`
2. Следующий запрос Zed попадает в `prepareOpenAIMessageInput(messages, ..., model)`, которая проверяет флаг
3. Если флаг установлен: `applyForceFolding()` сжимает историю — первые 5 сообщений (оригинальная задача), последние 10 (текущее состояние), середина суммируется как статистика (`summarizeDiscardedTools()`)
4. Результат складывается через `buildStatelessTranscript` → компактный user message для свежего Qwen чата
5. Флаг сбрасывается — обычные запросы идут без сжатия

**Изменения в `src/api/routes.js`:**
- `forceResetModels` (Set) + `FORCE_FOLD_HEAD/TAIL` константы
- `applyForceFolding(messages, mappedModel)` — агрессивное сжатие истории
- `summarizeDiscardedTools(messages)` → `{role: "user", content: "[Previous agent work...]"}`
- `invalidateModelDefaultChat()` — добавлен `forceResetModels.add(model)`
- `prepareOpenAIMessageInput` — 4-й параметр `rawModel`, применяет Force Folding при наличии флага
- Оба обработчика (`/chat/completions`, `/v1/chat/completions`) передают `model`

## Сессия 18 (текущая — 2026-06-08 — Tool calling rewrite from Python fork)
**Проблема:** Zed Agent возвращает `"tool input was not fully received"` когда получает tool_call от прокси. Аргументы команды не доходят до терминала.

### Диагностика

1. **Qwen обрезает/ломает JSON** — наш `parseToolCallJson` брал весь текст между `{` и `}`, не балансируя скобки. Zed получал неполный payload → ошибка парсинга.

3. **Нет нормализации аргументов** — Python форк парсит и ре-строифицирует arguments через `json.dumps()`, гарантируя баланс скобок. У нас был сырой вывод Qwen.

1. **`parseToolCallParts(content)`** — новая функция, разделает смешанный ответ на `{ visible_text, raw_calls }`. Многоступенчатая стратегия:
   - Step 1: полное JSON parse
   - Step 2: поиск маркера `"tool_calls"`, обратный поиск обёртки `{` (ключ всегда ВНУТРИ объекта)
   - Step 2c: `_repairTruncatedBraces()` — балансировка скобок для обрезанного Qwen-ответа
   - Step 3: legacy fallback
2. **`normalizeToolCalls(calls)`** — парсит + строифицирует arguments через `JSON.parse()` → `JSON.stringify(null, null, 0)`. Гарантирует минификацию.
3. **Reasoning text в SSE** — `writeToolCallsSse()` принимает `visibleText`, отправляет reasoning chunk ДО tool_calls chunks. Идентично Python `stream_tool_calls`.

### Фикс: isToolFailure (`openaiUtils.js`)

3. Python-версия работает стабильно благодаря трём механизмам которых нет у нас:
   - Полные parameter schemas в tool prompt (у нас было только `compactJsonSchema`)
   - Anti-loop detection (`repeated_current_tool_calls`) — блокирует повторы с теми же аргументами
   - Explicit anti-repeat rules в prompt на русском языке
4. Без этих механизмов Qwen зацикливался: terminal call → результат от Zed → тот же call снова
5. `areAllToolsFailed()` тоже срабатывал неправильно — когда все tool results были ошибками, прокси отключал prompt injection и Qwen отвечал текстом вместо инструментов

### Корневая причина (сравнение Python vs Node)
Python шлёт в Qwen **полные schemas** каждого tool'а:
```python
{"tool_calls":[{"name":"terminal","description":"...","parameters":{"type":"object","properties":{}}}]}
```
Plus explicit rules: `"После результата tool/function НЕ повторяй тот же самый tool call с теми же arguments."`

Node.js отправлял только **компактные схемы без параметров** + только список имён:
```javascript
Available tools: skill_view, terminal, read_file, ... (without parameter schemas)
```
Qwen не знала ФОРМАТ параметров → гадания → одинаковые вызовы → зацикливание.

### Решение
1. **Rewrite `toolsToPrompt()`** — полная схема каждого tool с параметрами, идентично Python форку. Явные правила anti-repeat на русском + формат JSON. Qwen видит точные параметры инструмента.
2. **Rewrite `toolsToLightPrompt()`** — те же улучшения для agent-loop режима (model уже имеет tool results).
3. **Add `getRepeatedToolCalls(calls, messages)`** — из Python `repeated_current_tool_calls()`. Сравнивает сигнатуры (name + args) новых вызовов с выполненными. Если совпадение → блокирует loop.
4. **Add `getBlockedToolCalls(calls, messages)`** — из Python `blocked_after_failed_tool_calls()`. Блокирует file operations после "outside the project" ошибки.
5. **Интеграция в streaming path (routes.js)** — перед отправкой tool_call SSE проверяем anti-loop. Если повтор → отправляем текстовый ответ вместо зацикливания.

**Коммиты:**
- `662534a` — feat: port anti-loop detection and full tool schemas from Python fork

## Сессия 29 (текущая — 2026-06-09 — Fix: SSE tool_call delivery chunk splitting)
### Проблема
Зацикливание на `Search with regex` с ошибкой `tool input was not fully received`. Qwen генерирует правильный JSON tool_calls, но Zed Agent теряет данные при доставке через SSE.
Причина: `writeToolCallsSse` отправлял весь function.arguments одним data: чанком. При 92 tools и полных schemas — один line >10KB → Node.js TCP buffer отбрасывает часть → Zed не получает полные args.

### Корневая причина
- OpenAI-compatible клиенты ожидают **incremental partial deltas** для tool_calls (как для content)
- Мы отправляли full function object в одном чанке без разбиения
- Большие payloads теряются на SSE boundaries особенно при долгих agent-loop сессиях

### Решение
1. **Rewrite `writeToolCallsSse`** — разбиваем delivery на partial chunks:
   - Chunk 1: `{id, type, name}` (small metadata)
   - Chunk 2+: arguments по кусочкам ~500 chars (как OpenAI streaming)
2. Зed Agent накапливает deltas и восстанавливает полный tool_call корректно
3. **Debug logging** — логит количество чанков для мониторинга работы доставки

Эффект: eliminates `tool input was not fully received` loop. Теперь delivery robust к размерам arguments.

## Сессия 31 (текущая — 2026-06-09 — Memory leak mitigation: page pool GC + hard limit)
### Проблема
При длительном agent-loop (>100 вызовов) или нескольких одновременных клиентах память Chromium растёт монотонно. Пул страниц (`pagePool`) имел только soft limit на хранение (PAGE_POOL_SIZE=3), но:
- Нет ceiling на общее количество живых страниц (каждый concurrent request = новая страница)
- Мёртвые/idle страницы остаются в памяти до следующего health-check при getPage()
- При низком трафике протухшие страницы висят часами

### Решение
1. **`src/config.js`**: добавлены 3 новые константы:
   - `MAX_ACTIVE_PAGES` (default: 5) — hard limit на одновременно активные страницы
   - `PAGE_IDLE_TTL_MS` (default: 5 мин) — TTL для idle-страниц в пуле
   - `PAGE_GC_INTERVAL_MS` (default: 60 сек) — интервал периодического GC

2. **`src/browser/pagePool.js`**: полная переработка с memory leak mitigation:
   - Hard limit: `getPage()` блокируется при достижении MAX_ACTIVE_PAGES, ожидая освобождения
   - Счётчик `activeCount` отслеживает checkout/release
   - Каждая страница в пуле хранит `{ page, lastUsed }` для TTL-based eviction
   - `_runGC()`: закрывает страницы, простаивающие дольше PAGE_IDLE_TTL_MS
   - `startGC()` / `stopGC()`: управление периодическим таймером
   - Auto-start GC при загрузке модуля, `timer.unref()` для graceful shutdown
   - Расширенные stats через `getStats()`: activeCount, maxActivePages, idleTtlMs, gcIntervalMs

Эффект: память стабилизируется на предсказуемом уровне, деградация graceful (queue/wait вместо OOM crash).
