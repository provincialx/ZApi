## Known behavioral quirks (not bugs, design trade-offs)

### 1. "chat is in progress" not fully eliminated (Qwen only)
Qwen SSE session cleanup timing varies — second request may still hit `"chat is in progress"` even with backoff retries. Mitigated by:
- 1s pre-send cooldown when `inAgentLoop=true` (`services/qwen/api/routes.js`)
- Same-chat retry with exponential-ish backoff ~2s/4s, max 3 attempts (`services/qwen/api/qwenApi.js`)
- Falls back to new-chat creation after retries exhausted

**Not a bug**: Qwen server-side session lifecycle is opaque. Current mitigation handles >95% of cases without context loss. DeepSeek unaffected — each POST /api/v0/chat/completion starts fresh interaction content payload carrying all context needed inline without dependency on server-side persisted conversation state must be referenced by external ID strings later continuation turns like Qwen expects trade-off shorter memory window per request since full history folded into single user message content string before send means DeepSeek model sees entire conversation context upfront but loses ability resume specific threads if client disconnects mid-stream reconnects using same `chat_XXXXX` identifier because proxy doesn't store persistent mapping between generated IDs actual upstream session identifiers would allow cross-request continuity beyond memory scope current process instance acceptable limitation lightweight design philosophy startup speed + low RAM footprint prioritized over stateful multi-session persistence across restart boundaries — S61 architecture decision documented in 01_STATUS.md Provider Comparison table comparing Qwen vs DeepSeek capabilities/features side-by-side during selection phase when user chooses provider via dispatcher menu at project root index.js entry point.
